# Case 2: 项目健康度评估

## 案例背景

你是 TaskFlow 项目的产品经理，项目已经进行了 8 周。现在需要向老板汇报项目状态，评估项目健康度，识别潜在风险，并提出改进建议。

你手上有以下材料：
- 产品需求文档（PRD）
- 技术决策记录
- 8 周的会议纪要
- 100 条用户反馈数据
- 8 周的项目进度数据

## 学习目标

通过本案例，你将学会：
- 整合多个文档和数据源进行综合分析
- 识别项目中的关键问题和风险
- 分析决策的影响和效果
- 生成高管级别的项目报告
- 提出可执行的行动计划

## 项目背景

**TaskFlow** 是一个轻量级项目管理工具，目标用户是 10-50 人的中小型团队。项目计划 8 周完成 MVP 版本，团队 8 人。

**项目特点**：
- 原计划包含 AI 智能提醒功能（差异化卖点）
- 第 4 周遇到重大问题，做出艰难决策
- 最终成功交付，但过程曲折

## 使用步骤

### 1. 准备数据

如果需要重新生成项目文件：

```bash
cd case2-project-health/scripts
python generate_files.py
```

### 2. 启动 Claude Code

```bash
cd case2-project-health
claude
```

### 3. 文档理解

首先让 Claude 理解项目背景：

```
请阅读以下文档，了解 TaskFlow 项目：
1. docs/PRD.md - 产品需求文档
2. docs/decisions.md - 技术决策记录

请总结：
- 项目目标和核心功能
- 主要技术决策
- 是否有被废弃的决策
```

### 4. 进度分析

分析项目进度数据：

```
请分析 data/progress.csv 文件：
1. 项目完成度随时间的变化趋势
2. 团队速度（velocity）的变化
3. Bug 数量的变化趋势
4. 测试覆盖率的提升情况
5. 团队士气的变化

请生成可视化图表展示这些趋势。
```

### 5. 用户反馈分析

分析用户反馈：

```
请分析 data/user-feedback.csv 文件：
1. 反馈类型分布（功能问题、性能问题、界面体验、功能建议）
2. 问题优先级分布
3. 问题解决状态
4. 不同时间段的反馈趋势

找出最常见的问题和最受欢迎的功能。
```

### 6. 会议纪要分析

深入分析会议纪要：

```
请阅读 docs/meeting-notes/ 目录下的所有会议纪要（week01 到 week08），分析：
1. 项目的关键转折点
2. 重大决策及其背景
3. 团队遇到的主要挑战
4. 问题的解决过程

特别关注第 4 周的紧急会议，分析当时的决策过程。
```

### 7. 综合分析

进行综合分析：

```
基于所有文档和数据，请分析：
1. 项目健康度评分（1-10分）
2. 关键成功因素
3. 主要风险点
4. 决策质量评估（特别是放弃 AI 功能的决策）
5. 团队协作效果
```

### 8. 生成报告

生成高管报告：

```
请生成一份项目健康度评估报告，包括：
1. 执行摘要（200字）
2. 项目概况
3. 关键指标分析
4. 重大决策回顾
5. 风险与机会
6. 经验教训
7. 下一步行动计划

报告要专业、简洁，适合向高管汇报。
```

## 分析要点

### 项目时间线

- **第 1-2 周**: 顺利启动，团队士气高
- **第 3 周**: 开始遇到问题（AI 性能、Bug 增多）
- **第 4 周**: 危机爆发，做出艰难决策（放弃 AI 功能）
- **第 5 周**: 调整后恢复，性能优化成功
- **第 6-7 周**: 稳步推进，UAT 测试
- **第 8 周**: 成功交付，用户满意度 8.5/10

### 关键决策

1. **放弃 AI 功能** (第 4 周)
   - 背景：性能差、成本高、时间紧
   - 影响：保证按时交付，但失去差异化
   - 结果：正确决策

2. **性能优化优先** (第 3 周)
   - 背景：并发性能不足
   - 影响：投入大量资源
   - 结果：性能提升 4 倍

3. **聚焦核心功能** (第 4 周)
   - 背景：时间紧张
   - 影响：砍掉非核心功能
   - 结果：保证质量

## 常见问题

### Q: 文件太多，如何开始？
A: 先阅读 PRD 了解项目背景，再看进度数据了解整体趋势，最后深入会议纪要了解细节。

### Q: 如何判断决策是否正确？
A: 对比决策前后的数据变化，结合最终结果（按时交付、用户满意度）综合评估。

### Q: 报告应该多长？
A: 执行摘要 200 字，完整报告 1500-2000 字。重点是洞察，不是堆砌数据。

### Q: 可以用于真实项目吗？
A: 可以！这个分析框架适用于任何软件项目的健康度评估。
