先思考
大多数人觉得用 Claude Code 和其他 AI 工具时，第一件事就是输入指令（或是开口和 AI 对话）。但这很可能是你刚上手就会犯的最严重错误之一。真正该先做的，其实是先思考。

对你们当中有些人来说，这说起来容易做起来难。你可能没有多年的软件工程经验，没法独立理清这类问题的思路。针对这点，我有两条建议：
开始学习吧。要是你从来都不接触这类内容，哪怕每次只学一点点，也是在给自己设限。
2. 与 ChatGPT、Gemini、Claude 进行深入的来回互动，准确描述你想要搭建的内容，向 LLM 咨询系统设计层面的各类可行方案，最终双方敲定一套解决方案。你和 LLM 应当互相提问，而非单向输出。
这适用于所有场景，就连整理邮件摘要这类细碎小任务也不例外。在让 Claude 开发新功能前，先构思好整体架构；让它重构代码前，先明确最终应该达成的效果；让它调试 bug 前，先理清你对这个问题实际掌握的信息。你在规划阶段提供的信息越充分，最终得到的结果就越好——毕竟输入越好，输出自然也就越好。
这点确实成立：先思考再打字，产出的效果远比先直接敲字再指望 Claude 理清思路要好得多。
这就说到我接下来要讲的架构相关内容了。尤其是在软件工程领域，架构有点像只给别人最终成果，却半点不透露其他信息。这会在实现最终成果的路径上留出极大的灵活空间，而这恰恰就是 AI 生成代码的核心问题所在。如果你只是笼统地说“帮我做一个身份验证系统”，而非“使用现有的 User 模型搭建邮箱/密码认证功能，将会话存储在 Redis 中并设置 24 小时过期时间，还要添加中间件来保护/api/protected 路径下的所有路由”，就能看出两者的差别了。
按两次 Shift+Tab 键，就能进入规划模式了。听我一句劝，这只需要花你 5 分钟，但之后能帮你省下好几个小时的调试时间。
克劳德.md
CLAUDE.md 是一个 Markdown 文件。Markdown 是一种人工智能模型处理起来极为得心应手的文本格式，尤其是克劳德（Claude），它在处理这类格式上的表现比我测试过的绝大多数其他模型都更出色。
启动克劳德代码会话后，克劳德首先会读取你的 CLAUDE.md 文件。文件里的每一条指令都会决定克劳德处理你项目的方式，这本质上就是克劳德每次对话前都会先看的入门指南。
大多数人要么干脆完全无视它，要么往里面塞一堆垃圾内容，结果反倒让 Claude 变得更糟而非更好。存在一个临界点，信息过多或过少都会让模型输出效果变差。
真正重要的是：
别太长。Claude 一次只能稳定处理约 150 到 200 条指令，而 Claude Code 的系统提示已经用掉了大概 50 条。你每新增一条指令，都会和它争抢上下文注意力。要是你的 CLAUDE.md 写得像长篇小说，Claude 就会开始随机忽略部分内容，你根本不知道它漏了哪些。
针对你的项目做定制化调整。不用解释 components 文件夹是什么，Claude 懂 components 是什么。跟它讲那些真正有用的冷门操作，比如关键的 Bash 命令。所有属于你工作流程的内容都要加进去。
要告诉它为什么，而不只是做什么。克劳德在这点上还挺像人类的。你要是给出指令背后的原因，克劳德的执行效果会比你只吩咐它做什么要好得多。「启用 TypeScript 严格模式」还行，但「启用 TypeScript 严格模式，因为我们之前栽在隐式 any 类型上出过生产环境 bug」效果更佳。说明原因能帮克劳德获得你没预料到的、用于做决策判断的上下文。你会惊觉这招居然这么好用。
持续更新它。干活的时候按#键，Claude 就会自动把指令添加到你的 CLAUDE.md 文件里。要是你发现自己两次都在纠正 Claude 同一件事，那就说明该把这个内容加到这份文件里了。久而久之，你的 CLAUDE.md 就会变成一份能如实体现你代码库运作方式的活文档。
差劲的 CLAUDE.md 就像是写给新员工的官方文档。靠谱的 CLAUDE.md 则是你明知道自己明天就会失忆时留给自己的备忘笔记。
上下文窗口的局限性
举个例子，Opus 4.5 拥有 20 万令牌的上下文窗口。但大多数人没意识到的是：这个模型在你用满 100%容量之前就开始出现性能衰减了。（这取决于你是通过 API 还是桌面应用使用）
当上下文使用率处于 20%到 40%区间时，哪怕不算特别明显，模型输出的质量也会开始下滑。要是你曾遇到过 Claude Code 先进行上下文压缩，之后却依然给出糟糕输出的情况，那原因就是这个。其实在压缩操作执行前，模型的性能就已经下降了，而压缩并不能神奇地恢复输出质量。（输入/compact 即可执行压缩）
你发送的每一条消息、Claude 读取的每个文件、它生成的每一段代码，还有每一项工具返回的结果——所有这些都会不断累积。一旦输出质量开始下滑，更多的上下文反而会让情况变得更糟，而非更好。所以下面这些方法，能帮你避开糟糕的上下文困境。
给你的对话划定范围。每个功能或任务单独开一个对话。别用同一个对话同时搭建认证系统和重构数据库层。上下文会串味，Claude 会搞混的。我知道至少有一位正在看这条推文的人犯过这个错。
学会善用外部存储。如果你在处理复杂项目，让 Claude 把计划和进度写入实体文件（我一般用 SCRATCHPAD.md 或者 plan.md）。这些文件会在会话间保留下来。等你第二天回来继续时，Claude 可以读取这些文件，接着上次的进度继续，不用从零开始。顺带一提：如果你有文件层级体系，把这类文件放在最顶层，就能让它适配你要开发的所有任务或功能。
复制粘贴重置法。这是我常用来救场的小技巧。当上下文变得臃肿冗余时，我会先从终端里复制所有重要内容，运行/compact 生成摘要，再用/clear 彻底清空上下文，最后只把关键内容粘贴回去。这样既能保留核心信息，又能获得全新的上下文，比让 Claude 在质量下降的上下文里硬扛要好得多。
懂得何时清空对话。要是聊天跑题了，或是攒了一堆无关的上下文，直接用/clear 就能从头再来。硬着头皮理顺混乱的思路远不如这样做。克劳德还会保留你的 CLAUDE.md 文件，所以你不会弄丢项目的上下文信息。尽管听起来有点反直觉，但十有八九，用/clear 其实比不用更好。
实用的心智模型：Claude 是无状态的。除了你明确提供的内容外，每一次对话都从零开始。据此规划即可。
提示词就是一切
大伙花几周时间学习框架和工具，却半点儿时间都不肯花在琢磨怎么跟真正生成代码的家伙沟通。
提示词工程可不是什么神秘玄学。它大概是现存最基础的沟通形式了。而和任何沟通方式一样，清晰明确永远比含糊不清能带来更好的效果。每一次，都是如此。
真正有用的是：
明确说明你的需求。“构建一个认证系统”会给 Claude 太多发挥空间，结果往往不尽如人意。“基于现有 User 模型搭建邮件/密码认证系统，将会话存储在 Redis 中，并添加中间件来保护/api/protected 路径下的所有路由”，这样才能给 Claude 明确的目标。即便如此，这也仍算不上完美。
告诉它不要做什么。Claude 有一些通病，尤其是 Claude 4.5，总爱过度设计——多出不必要的文件、画蛇添足的抽象层，还会给你加压根没要求过的灵活性。如果你想要极简的成品，就说：「保持简洁，不要添加我没要求的抽象层。如果可以的话，尽量只用一个文件。」另外，一定要仔细核对 Claude 生成的内容，不然你可能会背上技术债务，尤其是当你只想做个超简单的东西，结果它却为一个本可以用几行代码搞定的任务生成了 12 个不同的文件。
你得记住：AI 是为了帮我们提升效率而生，而非彻底取代我们，尤其是在专业软件工程愈发重要的当下。Claude 照样会出错，我敢说哪怕它之后会越来越完善，也依然会不断犯错。所以学会识别这些错误，其实能帮你解决不少问题。
先讲清原因背景。「我们需要它快，因为每个请求都会运行它」这句话会改变 Claude 处理问题的思路。「这是个我们会丢弃的原型」则会调整合理的权衡取舍方向。至于你没提到的约束条件，Claude 根本没法猜透。
记住：你产出的一切都源于输入。要是你的成品拉胯，那说明你的输入本身就不行，这点根本没辙。
输入垃圾，输出垃圾
人们拿到糟糕结果时，总爱怪模型。“Claude 不够聪明”或是“我得换个更好的模型”。
认清现实：你就是菜。要是你用 Opus 4.5 这么好用的模型却得到了糟糕的输出，那说明你的输入和提示词有问题，没跑了。
模型本身确实很重要，而且真的重要得多。但在当下，模型质量已经是基本标配了。瓶颈几乎永远都在人类这边：你如何构建提示词、如何提供上下文，又如何清晰地传达自己真正想要的东西。
如果你总得到糟糕的结果，解决之道并非切换模型，而是要精进以下方面的能力：
如何撰写提示词。具体 > 模糊。约束条件 > 开放式表述。示例 > 单纯描述。
如何构建请求结构。将复杂任务拆分为多个步骤。在实施前就架构方案达成共识。审查输出结果并迭代优化。
你该如何提供上下文？要想出色完成任务，Claude 需要知晓哪些信息？你又有哪些 Claude 看不到的预设假设？
话虽如此，不同模型之间确实存在实质性差异：
Sonnet 更快也更划算。它非常适合路径清晰的执行类任务——比如编写样板代码、按照具体规划重构，或是在已经敲定架构方案的前提下实现新功能。
Opus 速度更慢、成本也更高。它更适合用于复杂推理、规划，以及需要 Claude 深入权衡利弊的任务。
一套好用的工作流：先用 Opus 规划架构、敲定设计方案，再切换到 Sonnet（在 Claude Code 里按 Shift+Tab）来落地开发。这得看具体任务而定，有时候你也可以用 Opus 4.5 来完成开发。但如果你是通过 API 使用面板来操作的话，那可得做好卖肾付账单的心理准备啊。你的 CLAUDE.md 文件能让两个模型遵循完全一致的约束条件，所以模型交接会非常丝滑。
MCP、工具与配置
克劳德的功能多到离谱！MCP 服务器、钩子、自定义斜杠命令、Settings.json 配置、技能、插件。
你用不着全都用上，但真的该都试试、多摸索摸索。毕竟要是不肯尝试折腾，你多半就是在白白浪费时间和金钱。我跟你打包票，Claude 至少会推出一项你从没听过的新功能，只要你关注 Claude Code 的创始人鲍里斯，就能了解到这些新内容。
MCP（模型上下文协议）可以让 Claude 连接外部服务，比如 Slack、GitHub、数据库以及各类 API。要是你总需要把别处的信息复制粘贴到 Claude 里，那大概率有对应的 MCP 服务器能自动完成这项工作。现在有超多 MCP 应用市场，就算暂时没有你需要的工具，MCP 本质上就是获取结构化数据的一种方式，你完全可以为当前暂未上架的所需工具搭建专属的 MCP 服务器。不过说实话，你几乎很难找不到对应的 MCP，要是真能找到一个不存在的，我反倒会大吃一惊。
钩子功能可以让你在 Claude 修改代码前后自动运行代码。想让 Prettier 在 Claude 处理过的每个文件上都生效？用钩子就行。想在每次编辑后都进行类型检查？同样用钩子。这样就能立刻发现问题，而非让问题越积越多。这其实也有助于减少技术债务。如果你每写满一千行代码就设置一个特定钩子，这个安全功能就能帮你清理代码。在 Claude 审核你的 PR 时，这应该会大有用处。
自定义斜杠命令本质上就是你反复使用的提示词，被封装成了命令格式。你可以创建一个 .claude/commands 文件夹，添加包含你提示词的 Markdown 文件，之后就能通过 /commandname 来运行它们了。如果你经常重复执行调试、代码审查、部署这类任务，直接把它做成斜杠命令就行。
如果你办了 Pro Max 套餐（我每个月付 200 美元），不如把 Claude 能提供的所有功能都试一遍？看看哪些好用哪些不好用，反正你都花钱了。
说真的：要是第一次尝试没成，别灰心。这类模型基本每周都在迭代优化。一个月前还不管用的东西，现在说不定就能用了。作为早期尝鲜用户，就得保持好奇心，反复去测试这些东西。
当克劳德卡壳时
有时候 Claude 就是会陷入循环：反复尝试同一件事，失败了重来，再失败接着试，就这么一直耗下去。要不就是信心满满地写出完全错误的代码，你得花二十分钟去跟它解释哪里出问题了。
遇到这种情况时，本能反应就是继续加码：加更多指令、更多修正、更多背景信息。但实际上更好的选择其实是彻底改换思路。
先从简单的来——清空对话吧。之前累积的上下文可能会搞乱对话，输入/clear 就能获得全新的开局。
把任务简化一点。要是 Claude 卡在复杂任务上，就把它拆成一个个小部分。先让每个小部分都跑通，再把它们整合起来。但说实话，如果 Claude 连复杂任务都搞不定，那就说明你的规划模式不够完善。
别光说不练，直接演示。要是克劳德老是误解你的需求，那就自己写一个最小示例。“这就是预期的输出效果。现在把这个模式套用在其余部分上。”克劳德非常擅长理解成功标准是什么样的，也能切实照着标准执行，还清楚什么才是优质示例。
发挥创意，换个角度试试。有时候你给问题设定的框架和 Claude 的思考方式不太适配。换个表述重构问题——比如把「处理这些状态转换」改成「以状态机的方式实现它」——就能打开局面，推进进度。
这里的元技能是及早察觉自己陷入了循环。要是你已经把同一件事解释了三遍，克劳德还是没搞懂，那再多解释也没用，换个思路就好。
构建系统
能从 Claude 中获得最大价值的用户，根本不会只用它来完成一次性任务。他们正在搭建将 Claude 作为组件的整套系统。但 Claude Code 可比这好用太多了——它带有用于无头模式的-p 参数，无需进入交互界面就能运行你的提示词并输出结果。这意味着你可以为它编写脚本，将输出通过管道传输到其他工具中，和 bash 命令组合使用，还能将其集成到自动化工作流里。
企业正在用这个工具自动完成公关审核、客服工单自动回复、日志自动记录以及文档更新。所有操作都会被记录在案，可审计，并且会根据实际效果随时间不断优化。
飞轮效应：克劳德出错时，你查看日志，优化 CLAUDE.md 文件或是相关工具，下次克劳德就能表现得更好。这种效果会不断叠加。不过现在我已经在推进让克劳德自行优化它自己的 CLAUDE.md 文件了。经过数月的迭代，这类系统会比刚上线时好用得多——模型本身没变，只是配置优化得更到位了。
要是你只交互式地使用 Claude，那可就错失了不少本该到手的价值。不妨想想你的工作流里，有哪些环节可以让 Claude 在你不用盯着的情况下自动运行。
太长不看
先想好再打字。提前规划远比贸然开口效果要好得多。
CLAUDE.md 就是你的核心抓手。内容要短要具体，讲清原因，还要持续更新。单靠这一个文件就能影响所有交互。
上下文只会丢 30%，不是全没了。可以用外部记忆功能，捋顺对话上下文，别怕用复制粘贴重置法清掉重启。
架构比什么都重要。你不能跳过规划环节。要是不先把整体结构想清楚，最终的产出效果肯定很差。
输出全靠输入。要是用不错的模型却得不到理想结果，那说明你的提示词没做好。得多提升下沟通技巧。
多试试工具和配置。MCP、钩子、斜杠命令。要是你付费用了 Pro Max 版，那就把所有东西都试个遍。哪怕第一次没成功，也要保持好奇心。
卡壳时就换个方法。别死循环。理清、简化、展示、重构。
构建系统，而非一次性脚本。无头模式、自动化、随时间记录优化进展。
如果你在用 Claude 开发——不管是做个人项目还是上线生产系统——这些要点将决定你到底是在和这个工具较劲，还是能和它顺畅配合。
